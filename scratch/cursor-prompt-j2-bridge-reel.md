## Scope of this PR

Single PR, ~7 new files + 2 light extensions. **Do not modify** the existing `lib/creatives/remotion/{provider,shared}.ts` still-render code path — this PR rides alongside it. Provider-interface refactor (task #24, `render()` userId arg) is a separate PR and explicitly out of scope here.

### New files

1. **`src/remotion/compositions/PhotoReelStatic.tsx`** — generic photo-reel composition, 1080×1920 @ 30fps. Inputs:
   ```ts
   interface PhotoReelStaticProps {
     photos: string[];        // Supabase Storage URLs in render order
     framesPerPhoto: number;  // 7 for half-beat @ 130BPM/30fps
   }
   ```
   Per-photo layout:
   - **Background layer:** same photo, `object-fit: cover` (1280×1920), Gaussian blur 50px, dark overlay `rgba(0,0,0,0.35)`.
   - **Foreground layer:** same photo, `object-fit: contain` (1080×1620 inside 1080×1920 frame), centered vertically.
   - **Motion:** foreground scales `1.00 → 1.04` linearly across its `framesPerPhoto` window (subtle Ken Burns; softens the 0.23s cut rate).
   - **Hard cuts** between photos. No cross-dissolves.
   - Use `<AbsoluteFill>` + `<Sequence from={i * framesPerPhoto} durationInFrames={framesPerPhoto}>` per photo. Use `interpolate` + `useCurrentFrame` for the zoom.
   - Use `<Img src={url} pauseWhenLoading />` (Remotion's `<Img>`, not raw `<img>`). This ensures rendering waits for asset load and prevents frame-skip glitches.

2. **`src/remotion/index.tsx`** — extend registry to add `PhotoReelStatic` alongside existing `FourTfCityStatic`. `durationInFrames` calculated from `inputProps.photos.length * inputProps.framesPerPhoto`. Default inputProps load from `scratch/j2-bridge-render-input.json` if present (generated by the upload script — step 3 below); otherwise empty array (composition no-ops).

3. **`scripts/upload-reel-photos.ts`** — one-shot Node script (run locally during PR setup):
   - Reads `scratch/j2-bridge-manifest.json`
   - Reads source photos from `process.env.REEL_SOURCE_DIR` (Matas sets to `~/Documents/OFF Pixel/Junction 2/Melodic/Photos/Bridge`)
   - For each photo: sharp resize to 1080×1620, JPEG quality 82, upload to Supabase Storage at `campaign-assets/remotion-source/j2-melodic-bridge-2025/bridge-{NN}.jpeg`
   - Writes `scratch/j2-bridge-render-input.json` with the ordered array of public URLs + render specs:
     ```json
     {
       "compositionId": "PhotoReelStatic",
       "inputProps": {
         "photos": ["https://...bridge-01.jpeg", "..."],
         "framesPerPhoto": 7
       }
     }
     ```
   - Uses `@supabase/supabase-js` with `SUPABASE_SERVICE_ROLE_KEY` from `.env.local`. Add `sharp` as a devDependency if not already present.
   - Idempotent: skip uploads if target already exists in bucket (use `list()` to check). Allow `--force` flag to re-upload.
   - Run with: `npx tsx scripts/upload-reel-photos.ts`

4. **`app/api/admin/remotion/render-reel/route.ts`** — POST route, cookie auth, `maxDuration = 600`:
   - Reads `scratch/j2-bridge-render-input.json` (committed via the upload script's output)
   - Calls Remotion `renderMedia` (NOT `renderStill`) — codec `h264`, fps 30, audio `null` (no audio), pixel format `yuv420p`, CRF 23, image format `jpeg`
   - Uploads resulting MP4 to Supabase Storage at `campaign-assets/remotion-renders/{userId}/{renderId}.mp4`
   - Returns `{ jobId, assetUrl, durationSec, sizeBytes }` with a 7-day signed URL
   - Reuses bundle from `.remotion/bundle` (same as still route — bundled at build time via existing `scripts/bundle-remotion.ts`)
   - Errors throw `CreativeProviderDisabledError` if `FEATURE_REMOTION` is off (use existing helper from `lib/creatives/types.ts`)

5. **`app/admin/render-reel/page.tsx`** — minimal admin page:
   - Auth-gated (same pattern as existing `/admin/render-test/page.tsx`)
   - Single button: "Render J2 Bridge reel"
   - Calls the route, shows progress (spinner), embeds the resulting MP4 via `<video src={assetUrl} controls />` on success
   - Shows `{ durationSec, sizeBytes }` and the signed URL
   - Add a Download link

6. **`scripts/bundle-remotion.ts`** — extend the existing prebuild bundler script (if it doesn't already auto-detect all compositions) to include `PhotoReelStatic`. Should be a no-op extension if the bundler already crawls `src/remotion/index.tsx`.

7. **`lib/creatives/remotion/__tests__/photo-reel.test.ts`** — unit tests:
   - `PhotoReelStaticProps` type accepts the expected shape
   - `durationInFrames = photos.length * framesPerPhoto` (test with `photos: [...64 fakes], framesPerPhoto: 7` → expect `448`)
   - Component renders without throwing when given empty photos array (no-op safety)
   - Mock React render; do NOT call the full bundler in tests (slow + flaky)

## Non-negotiables

1. **Do not modify** `lib/creatives/remotion/provider.ts` — that's the still-render provider, task #24 is its separate PR.
2. **Do not bake audio** into the render. `audioCodec: null` on `renderMedia`. The user syncs audio in CapCut post-render.
3. **Do not commit the source photos** to the repo. They go to Supabase Storage only.
4. **Do not commit** `scratch/j2-bridge-render-input.json` if it contains secrets — it's just public URLs + numbers, so safe to commit. Verify before commit.
5. **`FEATURE_REMOTION` gating** — render route MUST throw `CreativeProviderDisabledError` when flag is off. Same pattern as PR #531.
6. **No client-facing surface.** Both the render route and admin page are gated behind cookie auth + admin email check (same as existing render-test).
7. **`renderMedia` memory budget** — set Vercel function memory to 3GB explicitly via `export const config = { memory: 3008 }` in the route (Vercel Pro default may be lower).
8. **Khroma Collective credit** — NOT in the render itself, NOT in this PR. Matas handles attribution in the caption when posting.
