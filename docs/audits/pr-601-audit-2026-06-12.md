## 1. Error A root cause — "Ad set was not created in Meta — no metaAdSetId available"

### Where it fires
`app/api/meta/launch-campaign/route.ts`, Phase 4, the per-(creative × ad-set) loop:

```2979:2988:app/api/meta/launch-campaign/route.ts
    for (const internalAdSetId of assignedAdSetIds) {
      const metaAdSetId = adSetMetaIds.get(internalAdSetId);
      const adSetName = adSetNameById.get(internalAdSetId) ?? internalAdSetId;
      if (!metaAdSetId) {
        creativeEntry.adsFailed.push({
          adSetName,
          error: "Ad set was not created in Meta — no metaAdSetId available",
        });
        continue;
      }
```

`assignedAdSetIds` comes from `creativeToAdSetIds[creative.id]`, i.e. the keys in
`draft.creativeAssignments` (route.ts:2921 `invertAssignments(...)`). The lookup
`adSetMetaIds.get(internalAdSetId)` returns `undefined` for the 4 failing ad sets.

### Where `adSetMetaIds` is seeded for attach_adset
The attach_adset short-circuit seeds **one synthetic key per Phase-1-verified live ad set**:

```2256:2262:app/api/meta/launch-campaign/route.ts
    for (const liveAdSet of liveAdSets) {
      const synthKey = attachedAdSetKey(liveAdSet.id);
      adSetMetaIds.set(synthKey, liveAdSet.id);
      adSetLaunchResults[synthKey] = {
        launchStatus: "created",
        metaAdSetId: liveAdSet.id,
      };
```

`liveAdSets` = `attachedLiveAdSets` (everything Phase 1 verified) which is built only from
`attachAdSetIds` = `draft.settings.existingMetaAdSets` (route.ts:453–456, 1082–1127). Phase 1
fails the **entire** launch if any selected ad set is missing/archived/orphaned — so every ad set
in `existingMetaAdSets` is guaranteed to be seeded into `adSetMetaIds`.

### Conclusion
`adSetMetaIds` (keyed off the launch-time **ad-set selection**) and `creativeAssignments`
(the **assignment matrix** built in Step 6) have diverged: the matrix references 4 ad-set keys
whose IDs are **not in `existingMetaAdSets` at launch time**. The 4 quoted IDs
(`120243015152990342`, `120243015153320342`, `120243362488210342`, `120245007582100342`) exist
in Meta but were not part of the verified/seeded set — i.e. a **stale / divergent assignment
matrix** in cross-campaign `attach_adset` (operator changed the Step-1 ad-set multi-select after
Step 6 had already auto-assigned creatives to the larger set, leaving orphan keys in
`draft.creativeAssignments`).

### Is this caused by PR #601? — No.
PR #601's diff to `route.ts` is limited to three additions:
1. `applyPageInstagramOverridesToCreatives(launchCreatives, …)` (line 709) — returns a **new**
   array of shallow-cloned creatives, preserving `creative.id`. Does **not** touch
   `draft.creatives`, `draft.creativeAssignments`, `adSetMetaIds`, or `attachedLiveAdSets`.
2. The Phase 0e operator-override block (lines 744–753) — only runs for **existing-post**
   creatives; the test used NEW ads, so it didn't execute.
3. The Phase 3 re-apply + validator wiring (lines 2561–2599) — per-creative identity only.

None of these read or write the ad-set→metaAdSetId map. **Error A is a pre-existing
cross-campaign `attach_adset` matrix/selection-divergence bug**, not a regression from #601.

> ⚠️ Honesty flag on the "NEW since #601" premise. I cannot confirm from code that #601
> introduced Error A — the failing branch (2982) and its inputs are untouched by #601. The
> most plausible reason it became *visible* this run is that the launch reached Phase 4 with a
> stale matrix this time; it is not gated by #601. **Probe C (prod logs) is required to confirm
> whether `existingMetaAdSets` at launch actually had only 8 of the 12 matrix ad sets.**

### Fix shape (for later — do not implement yet)
- **One-line / defensive:** in Step 1's ad-set multi-select handler, prune
  `draft.creativeAssignments` keys that are no longer in `existingMetaAdSets` (reconcile on
  deselect). OR in Phase 4, downgrade the orphan-key case from a per-ad failure to a silent
  skip + single preflight warning, since the operator's *current* selection is the source of
  truth.
- Either way, add a Phase 2 log line dumping `[...adSetMetaIds.keys()]` vs the matrix keys so
  divergence is greppable.

## 2. Error B root cause — `@ionfestival` still on the wire (code=100 subcode=3858231)

### The wire-shape gate (the truth in the payload)
For NEW ads, `instagram_user_id` is emitted **only when `validatedIgActorId` is truthy**.
Identical gate across all three NEW-ad builders:

```621:630:lib/meta/creative.ts
  return {
    name: creative.name || "Ad Creative",
    // page_id (+ optional instagram_user_id) — assets are in asset_feed_spec.
    // NOTE: instagram_user_id, not instagram_actor_id (PR #569).
    object_story_spec: {
      page_id: creative.identity.pageId,
      ...(validatedIgActorId ? { instagram_user_id: validatedIgActorId } : {}),
    },
    asset_feed_spec: spec,
  };
```

(Same `...(validatedIgActorId ? {...} : {})` spread in `buildLinkCreative` ~L370 and
`buildVideoCreative` ~L433.) **When `validatedIgActorId` is falsy, `instagram_user_id` is
dropped entirely and Meta defaults `object_story_spec` to the Page's primary connected IG —
which for Page `145163125507298` is `@ionfestival`.** That is the rejection.

### Call chain — where `validatedIgActorId` comes from
`app/api/meta/launch-campaign/route.ts`, Phase 3:

```2587:2599:app/api/meta/launch-campaign/route.ts
      const rawIgActorId =
        creative.identity.instagramActorId ??
        creative.identity.instagramAccountId ??
        "";
      const creativePageId = creative.identity.pageId ?? "";
      const creativePageToken = pageTokenMap.get(creativePageId) ?? null;
      const validatedIgActorId = rawIgActorId
        ? await igValidator.validate(rawIgActorId, {
            pageId: creativePageId || undefined,
            pageToken: creativePageToken,
            operatorOverrideId: operatorOverride,
          })
        : null;
```

PR #601 **does** mutate the right field. Lines 2565–2567 set
`creative.identity.instagramActorId = operatorOverride` and `instagramAccountId = operatorOverride`,
so `rawIgActorId === 17841400485165463` (`@l_w_e`) — the correct value reaches the validator.
**The mutation is not the bug.** The bug is what the validator does with it.

### The validator silently discards the operator's pick
`lib/meta/ig-actor-validator.ts` — even on the operator-override fast-path, the chosen ID is
only returned if it is found in the **page-level list** OR the **BM-asset list**:

```163:183:lib/meta/ig-actor-validator.ts
      // ── Operator override: validate the chosen id on page-level list first ──
      if (isOperatorPick && pageId && pageToken) {
        const pageList = await fetchPageIds(pageId, pageToken);
        if (pageList.includes(igActorId)) {
          …
          return igActorId;
        }
      }
      if (isOperatorPick) {
        const bmList = await fetchBmIds();
        if (bmList.includes(igActorId)) {
          …
          return igActorId;
        }
      }
```

If `@l_w_e` is in **neither** list (or `pageToken` is `null`, so the page-level branch is
skipped), execution falls through Path 1/2 (same two lists again) to Path 3, which returns
`null`:

```208:220:lib/meta/ig-actor-validator.ts
      // ── Path 3: neither authorised — preserve b57a98e protection ───────────
      const reason = pageId && pageToken
        ? "both bm-asset and page-level lists empty"
        : pageId
          ? "bm-asset empty; no page token for page-level fallback"
          : "bm-asset empty; no pageId provided";
      console.error(
        `[ig-actor-validator] resolved via=none ` +
          `page=${pageId ?? "(none)"} ig=${igActorId} (${reason}) ` +
          `— creative will fall back to page-only identity`,
      );
      idCache.set(igActorId, null);
      return null;
```

`null` → `buildCreativePayload({ validatedIgActorId: undefined })` → `instagram_user_id` omitted
→ Meta uses `@ionfestival` → subcode 3858231.

### Two token-related ways the lists come back without `@l_w_e`
1. **BM list** (`fetchBmIds`) is fetched with `launchToken` (system user token):
   `createIgActorValidator(adAccountId, launchToken)` at route.ts:2532. If `@l_w_e` is not an
   IG asset attached to ad account `act_901661116878308` in the Business Manager, it won't appear.
2. **Page list** (`fetchPageIds`) needs `creativePageToken = pageTokenMap.get(pageId)` (route.ts:2592).
   `pageTokenMap` is populated from `resolvePageIdentity(pid, userFbToken)`. If page-token
   resolution returned `null` for `145163125507298`, the page-level branch is **skipped
   entirely** and only the BM list is consulted — and `GET /{pageId}/instagram_accounts` is
   exactly the endpoint that *would* surface a second linked IG like `@l_w_e`.

### Conclusion
**PR #601 mutates the correct field, but the value never survives the validator.** The override
is gated behind membership in lists that (a) require a page token that may be `null`, and (b)
reflect ad-account BM assets / the page's *resolvable* actors — which apparently do not include
`@l_w_e`. The "b57a98e protection" (omit unverified actors to avoid generic #100s) is correct for
**auto-resolved** IDs but is **wrong for an explicit operator pick**: it converts the operator's
deliberate choice into a silent fallback to the wrong identity. This is the
`feedback_layered_fix_pattern` failure mode — #601 added a fast-path *entry* to the validator but
left the disqualifying *exit* (Path 3 → `null`) in place.

### Why #600 didn't fix it
#600 only threaded overrides into the Phase 1.5 `pageToIg` map (engagement-audience resolution).
It never reached the creative `object_story_spec`, which is built in Phase 3 from
`validatedIgActorId`. Different code path entirely.

## 4. Recommended fix (for Matas + Claude to decide — NOT implemented)

### Error B — the decisive fix depends on Probe A
- **If Probe A ACCEPTS** (most likely): **trust the operator override and bypass list
  membership.** When `isOperatorPick` is true, return `operatorOverrideId` directly from the
  validator instead of falling through to Path 3 `null`. (~one line: in the
  `if (isOperatorPick)` block, after the two list checks, `return igActorId;` for operator picks
  rather than continuing.) Optionally guard it with a per-launch `validate_only` probe so we
  never knowingly send an ID Meta will reject — but do **not** let an empty BM/page list silently
  swap in `@ionfestival`.
- **If Probe A REJECTS:** this is not a code bug — `@l_w_e` is not an ads-eligible actor for this
  ad account. The fix is a Meta-side asset assignment (attach the IG to the BM/ad account), and
  the wizard should surface "this IG can't be used for ads from this account" instead of silently
  launching with the wrong one.

### Error A — independent of B
Reconcile `draft.creativeAssignments` with the live `existingMetaAdSets` selection (prune orphan
keys on deselect in Step 1), or downgrade the orphan-key Phase 4 case to a skip + preflight
warning. Add the Phase 2 key-dump log for observability.

### Cross-cutting
Both errors are surfaced only at Phase 4 with no preflight. Add a Phase 0 check that every
`creativeAssignments` key resolves to a seeded ad set, and (Error B) that every creative's chosen
IG identity is probe-validated, **before** any Meta mutation.

## 5. What we should have audited BEFORE PR #600

1. **Read the wire payload first, not the merge log**
   (`feedback_trust_payload_diff_over_recent_merge_premise`). The single most important fact —
   `instagram_user_id` is emitted only when `validatedIgActorId` is truthy (creative.ts:627) —
   was knowable from one file read. #596/#600/#601 all patched *upstream* of this gate without
   confirming the value clears it.
2. **`validate_only=true` probes before any code** (`feedback_validate_only_for_meta_field_bugs`).
   Probe 0 + Probe A would have told us in minutes whether `@l_w_e` is even an eligible actor —
   the deciding fact that determines whether this is a 1-line code fix or a Meta-asset fix.
3. **Trace the full call chain through the validator's *exit* paths, not just its entry.** #601
   added an operator-override *entrance* to the validator but left the `null` *exit* (Path 3)
   reachable for the operator's own pick. Classic `feedback_layered_fix_pattern` /
   `feedback_audit_first_when_layered_fixes_emerge`: three stacked patches, none of which followed
   the value to the wire.
4. **Separate the two failure domains earlier.** Error A (ad-set mapping) and Error B (IG
   identity) are unrelated; bundling them under "the LWE bug" masked that #601 couldn't have
   caused Error A.
