# Diagnosis — Dual-image launch fails with code=100 subcode=1772103

- **Date:** 2026-06-05
- **Branch:** `cursor/diag-561-dual-image-1772103`
- **Scope:** Diagnosis only — no fix applied. Grounded in actual payload diff.
- **Symptom:** Aberdeen WC26 traffic campaign, 3 ad sets (TOFU/MOFU/BOFU), dual
  image upload (4:5 Feed + 9:16 Story/Reel). Creative builds; all 3 `/ads`
  calls fail `code=100 subcode=1772103` "Select an Instagram account or
  Facebook Page". `ENABLE_MULTI_PLACEMENT_ASSETS` is OFF in prod.

## Verdict

**PR #561 is NOT the cause.** In flag-OFF state the `/adcreatives` payload for a
dual-image creative is **byte-for-byte identical** before and after #561 (proven
below). The real defect is that **new-ad link/image/video creatives are sent
page-only (no `instagram_actor_id`)**, introduced by commit **`b57a98e`
(2026-04-18)** — well before #561. With Instagram placements eligible (Stories /
Reels, which dual mode exists to serve), Meta rejects the `/ads` call because the
creative has no Instagram identity → 1772103.

The single-vs-dual distinction is **not in our code**: single 9:16 and dual
4:5+9:16 produce structurally identical page-only payloads. Whatever made an
earlier "single-asset" launch succeed was environmental (different placements /
objective / an existing-post creative / a launch predating `b57a98e`), not a
code path that attaches IG identity for single but not dual.

## Evidence 1 — #561 did not change the flag-OFF payload

Built `buildCreativePayload` + `sanitizeCreativeForStrictMode` (strictMode ON,
flag OFF) for the same dual-image creative at both commits:

- pre-#561  `c91da5a`
- post-#561 `ae77c92` (current main)

Resulting `/adcreatives` payload (identical at both commits):

```json
{
  "name": "Aberdeen WC26 — Dual Image",
  "object_story_spec": {
    "page_id": "PAGE_4THEFANS",
    "link_data": {
      "message": "Get your Aberdeen tickets",
      "link": "https://example.com/aberdeen",
      "name": "World Classic 2026",
      "call_to_action": { "type": "BOOK_NOW", "value": { "link": "https://example.com/aberdeen" } },
      "image_hash": "HASH_45"
    }
  },
  "degrees_of_freedom_spec": { "creative_features_spec": { /* opt-outs */ } }
}
```

The only post-#561 difference anywhere is an internal `assetFeedSpec` field on
the **sanitizer report object** (diagnostic metadata) — it is never sent to Meta.

`git diff c91da5a ae77c92 -- lib/meta/creative.ts` confirms the flag-OFF branch
is untouched: the new code is all behind `if (process.env.ENABLE_MULTI_PLACEMENT_ASSETS === "1")`
or inside the never-reached `buildMultiPlacementCreative`. The route changes in
#561 are **logging-only**.

## Evidence 2 — single and dual are structurally identical

Same builder (`buildLinkCreative`) for both. `object_story_spec` for each:

```
DUAL   : { page_id, link_data: { …, image_hash: "HASH_45"  } }   instagram_actor_id? false
SINGLE : { page_id, link_data: { …, image_hash: "HASH_916" } }   instagram_actor_id? false
structurally identical (ignoring the hash value): true
```

Both omit `instagram_actor_id`. So our code cannot explain "single worked, dual
didn't" — they are the same shape.

## Evidence 3 — the missing field and where it was removed

`code=100 subcode=1772103` is an **identity** error: the creative lacks an
Instagram identity for the eligible Instagram placements. The missing field is:

```
object_story_spec.instagram_actor_id   (the ads-authorised IG actor id)
```

It was deliberately removed in:

```
b57a98e  fix: remove instagram_actor_id from new_ad link/video creative payloads  (2026-04-18)
```

Before `b57a98e`, `buildLinkCreative` / `buildVideoCreative` did:

```ts
const igActor = resolveIgActorForPayload(creative.name, creative.identity, "buildLinkCreative");
if (igActor.source !== "none") {
  spec.instagram_actor_id = igActor.id;
}
```

`b57a98e` deleted that block (and the `resolveIgActorForPayload` helper) to dodge
a *different* error (`#100` when an unauthorised actor id was sent). That trade
left every new-ad creative page-only, which is fine for Facebook-only placements
but fails for Instagram placements with 1772103.

> Note: the launch log line "Page + IG · ID 1750802446345627" is a **red
> herring** — it comes from `identityMode` in the route, derived from
> `creative.identity.instagramAccountId`, NOT from what was sent. The payload
> omitted the IG id.

### Current lines (post-#561 `lib/meta/creative.ts`)

- `buildLinkCreative` page-only spec: ~347–360 (`const spec = { page_id, link_data }`, logs `instagram_actor_id = OMITTED`)
- `buildVideoCreative` page-only spec: ~405–412 (same shape for `video_data`)
- The new `buildMultiPlacementCreative` (flag-ON) has the **same defect**:
  `object_story_spec: { page_id: creative.identity.pageId }` only — so turning
  the flag on would not fix 1772103 either; it must be fixed in all new-ad paths.

## Reproduction

`lib/meta/__tests__/creative-ig-identity-regression.test.ts` asserts that a
new-ad image/video creative whose identity has an IG actor id includes
`object_story_spec.instagram_actor_id`. It is **RED on current main** (both
cases fail: "instagram_actor_id = OMITTED") and turns green when IG identity is
restored.

## Fix options (NOT applied — diagnosis only)

1. **Targeted re-add (recommended direction).** In `buildLinkCreative` /
   `buildVideoCreative` (and `buildMultiPlacementCreative`), set
   `spec.instagram_actor_id = creative.identity.instagramActorId` **only when it
   is a valid ads-authorised actor id** (the value resolved from
   `GET /{page-id}/instagram_accounts`, not the raw `instagramAccountId`). This
   restores IG-placement rendering while avoiding the `#100` that `b57a98e` was
   fighting (which came from sending an *unauthorised* id). Requires confirming
   `instagramActorId` is populated at launch time.
2. **Straight revert of `b57a98e`** (re-add `resolveIgActorForPayload` + the two
   call sites). Simplest, but reintroduces the original `#100` risk for accounts
   whose IG actor isn't authorised — only safe if `resolveIgActorForPayload`
   already validated authorisation. Files: `lib/meta/creative.ts`
   (`buildLinkCreative`, `buildVideoCreative`) + the actor-retry block in
   `app/api/meta/launch-campaign/route.ts`.

Recommendation: option 1, gated on a confirmed authorised `instagramActorId`,
with the regression test above as the acceptance gate. Decision needed before
implementing because of the `#100`/`1772103` trade-off `b57a98e` documented.

## One-line takeaway

Page-only new-ad creatives (from `b57a98e`, not #561) carry no Instagram
identity, so Instagram placements are rejected with 1772103; restore a validated
`instagram_actor_id` to the new-ad builders.
